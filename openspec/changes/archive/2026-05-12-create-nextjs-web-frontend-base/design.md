## Context

本次变更目标是在现有仓库中建立可扩展的 NextJS 工具网站基础框架，作为后续多个工具功能的统一承载层。当前项目尚无稳定的前端工程基线与工具站信息架构，需要先统一目录组织、页面结构和工程规范，避免后续工具接入出现重复建设。

## Goals / Non-Goals

**Goals:**
- 提供可运行的 NextJS（App Router）基础应用骨架。
- 建立清晰的前端目录结构与基础页面布局。
- 建立 TypeScript、ESLint 与 npm scripts 的统一工程基线。
- 建立工具网站首页与工具详情页的基础信息架构，便于后续新增工具。
- 为后续 API 对接与环境配置提供预留入口。

**Non-Goals:**
- 不实现具体业务页面与业务数据流。
- 不在本次引入状态管理、国际化、组件库等扩展能力。
- 不定义后端接口契约与鉴权机制。
- 不在本次实现具体工具算法与工具业务逻辑。

## Decisions

- 采用 NextJS + App Router 作为基础框架，优先获得文件路由、服务端渲染能力与后续扩展灵活性。  
  备选方案是 Vite + React Router，但在路由约定、服务端能力与生态一致性上不如 NextJS 适合作为统一基线。
- 采用 TypeScript 作为默认语言，保证后续模块边界和接口定义可维护。  
  备选方案是纯 JavaScript，但在多人协作和长期演进中类型约束不足。
- 采用最小可用工程配置（ESLint + 基础脚本），避免在框架期引入过重工具链。  
  备选方案是一次性引入完整工程体系（测试、提交规范、样式方案等），但会增加初始复杂度。
- 目录按“路由层、组件层、工具层、资源层”拆分，优先可读性与低认知成本。  
  备选方案是按业务域预拆分，但当前没有业务模块，过早分层会造成空结构。
- 路由结构采用“工具首页 + 工具详情页（动态段）”的最小信息架构，支持工具逐步扩展。  
  备选方案是仅保留单页模板，但会在新增第二个工具时产生路由与导航重构成本。

## Risks / Trade-offs

- [风险] 初期目录结构过于通用，后续业务快速增长时可能需要调整。  
  → 缓解：在任务中保留约定说明，后续按能力演进逐步引入按域拆分。
- [风险] 仅提供最小工程基线，可能无法覆盖团队全部质量诉求。  
  → 缓解：先确保可运行与一致性，再通过独立变更补充测试与格式化规范。
- [权衡] 当前不引入复杂依赖，短期开发体验简单，但高级能力需后续增量接入。  
  → 缓解：在配置中保留可扩展入口，避免二次重构成本。
- [风险] 工具页面数量增长后，导航与分类策略可能发生变化。  
  → 缓解：先抽象工具元信息结构（名称、描述、路由），后续可平滑升级为分组与搜索。
