## Why

IceBox（冰盒）需要同时支持工程师本地快速迭代、开发环境联调验证以及生产环境稳定发布。若三套环境混用同一套启动方式或配置入口，易出现「本地能跑、容器不能跑」或开发与生产配置串线的问题。现在明确区分本地（无 Docker）、开发（Docker）与生产（Docker），可降低环境差异带来的故障成本。

## What Changes

- 定义并文档化三种运行层级：本地环境、开发环境、生产环境，各自有清晰的配置与启动路径。
- 本地环境：仅使用 Node/npm（或等价本地工具链）启动与调试，不以 Docker 作为前置条件。
- 开发环境：提供 Docker 镜像与编排（或等价容器方案），用于在服务器上以独立目录部署开发实例。
- 生产环境：提供面向生产的 Docker 镜像与编排，与开发环境在目录、端口、密钥与镜像标签上可区分。
- 增加环境变量与机密的约定示例（不提交真实密钥），区分各环境使用的变量集合。
- 在仓库中增加可复用的部署脚本或说明，使「按目录区分开发/生产部署」成为默认操作方式。

## Capabilities

### New Capabilities

- `local-runtime-no-docker`：定义本地环境 SHALL 在无 Docker 前提下可完成安装、启动与基础验证的能力。
- `docker-deployment-development`：定义开发环境使用 Docker 部署时的目录隔离、编排与配置注入能力。
- `docker-deployment-production`：定义生产环境使用 Docker 部署时的镜像、编排与安全基线能力。
- `multi-tier-environment-model`：定义本地、开发、生产三层的命名、配置边界与禁止混用的规范能力。

### Modified Capabilities

无（本次为新增部署与环境模型能力，不改变既有前端页面与路由的规范语义。）

## Impact

- 受影响代码：新增 `Dockerfile`、Compose 文件、部署脚本及环境示例文件；可能调整 `README` 与 `docs`。
- 受影响依赖：容器构建可能引入多阶段构建与运行时镜像选型。
- 受影响系统：CI/CD、服务器目录布局、运维人员的发布与回滚流程。
- API 影响：无直接 API 契约变更；仅可能通过环境变量影响前端请求的基地址配置。
