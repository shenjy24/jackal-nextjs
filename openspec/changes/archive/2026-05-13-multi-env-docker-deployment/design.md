## Context

IceBox（冰盒）为 NextJS 前端应用，当前已在仓库根目录提供本地 `npm` 脚本。用户要求明确三套环境：本地（无 Docker）、开发（Docker）、生产（Docker），且服务器上通过**不同部署目录**区分开发与生产，而非仅靠启动参数混在同一目录。

## Goals / Non-Goals

**Goals:**

- 本地环境：工程师克隆仓库后，仅依赖 Node/npm 即可完成安装、开发启动与基础质量检查，无需安装 Docker。
- 开发环境：在服务器固定目录（例如 `.../icebox-dev`）内，使用 Docker Compose（或等价编排）拉起服务，配置与端口与生产隔离。
- 生产环境：在服务器另一固定目录（例如 `.../icebox-prod`）内，使用 Docker 构建与运行，具备可重复构建与简单回滚路径。
- 三套环境使用不同的环境变量文件或注入方式，避免密钥与 API 基地址串用。

**Non-Goals:**

- 不在本次定义具体云厂商（如 K8s、Serverless）的完整生产拓扑。
- 不在本次实现后端服务或数据库容器（若未来需要可另起变更）。
- 不强制统一「开发也禁止本地 Docker」——仅要求「本地不以 Docker 为前置条件」。

## Decisions

- **目录即环境边界**：开发与生产在服务器上使用两个独立工作目录，各自包含各自的 `.env`（或 compose 的 `env_file`）、`docker-compose.yml` 与数据卷挂载点。  
  备选方案为单目录多 profile，但不符合「目录区分」的运维诉求。

- **本地使用 `npm run dev`**：与现有 NextJS 生态一致；CI 与本地共用 `lint`、`build` 脚本作为质量基线。  
  备选为本地也用 Docker 模拟，但会增加本地机器负担且与需求冲突。

- **容器镜像采用多阶段构建**：一阶段安装依赖并执行 `next build`，运行阶段仅包含生产依赖与构建产物，减小攻击面与镜像体积。  
  备选为单阶段镜像，构建慢且运行时携带 devDependencies。

- **开发容器可挂载源码卷（可选）**：便于热更新；生产镜像不挂载源码，仅运行构建结果。  
  若团队更偏好「开发镜像与生产一致」，可在 Open Questions 中收敛为纯构建产物运行。

- **环境变量命名**：使用 `NEXT_PUBLIC_*` 暴露浏览器侧配置，服务端密钥仅通过运行时注入，不写入镜像层。  
  备选为构建期 `ARG` 注入敏感信息，存在泄漏风险故不采用。

## Risks / Trade-offs

- [风险] 开发与生产 compose 分叉导致「开发能跑、生产不能跑」。  
  → 缓解：抽取公共 `Dockerfile`，差异尽量仅在 compose 覆盖层与环境文件。

- [风险] Windows 与 Linux 路径或行尾差异影响脚本。  
  → 缓解：部署脚本以服务器 Linux 为主文档说明；仓库已配置 `.gitattributes`。

- [权衡] 开发环境若挂载源码，行为与生产略有差异。  
  → 缓解：发布前必须在生产镜像路径执行 `build` 与冒烟验证。

## Migration Plan

1. 在仓库增加 `Dockerfile` 与 `deploy/dev`、`deploy/prod`（或等价命名）目录下的 compose 与示例环境文件。
2. 在开发服务器创建独立目录，拉取指定分支，复制示例 env 为实际 env（不入库），执行开发 compose 启动。
3. 在生产服务器创建独立目录，拉取发布标签或主分支，配置生产 env，执行 `docker compose build && up -d`。
4. 回滚：保留上一版镜像 tag 或上一构建目录，执行 `docker compose up` 指回旧镜像，或切换目录软链（若采用）。

## Open Questions

- 开发环境是否需要「源码热挂载」还是「与生产一致仅运行构建产物」？
- 反向代理与 TLS 终止由本仓库 compose 管理，还是由运维在主机 Nginx/Caddy 统一管理？
- 生产环境期望的对外端口与域名路径前缀（若有）是否已确定？
